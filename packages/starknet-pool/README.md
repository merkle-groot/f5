# starknet-pool

Destination-side Cairo shielded pool — the Starknet analog of `packages/contracts/src/contracts/L2/L2PrivacyPool.sol`, receiving Mode-3 withdrawals bridged from the L1 pool (`PrivacyPool._bridgeStarknet`).

📄 **Full write-up (architecture, live Sepolia run, bugs found, reproduce steps):** [`docs/l1-to-starknet-bridge.md`](docs/l1-to-starknet-bridge.md)

## Layout

- `src/pool.cairo` — the pool contract (`StarknetPrivacyPool`): StarkGate `on_receive` intake after token credit, backing invariant, LeanIMT, and `withdraw` with Poseidon `context` + Garaga proof verification.
- `src/hashing.cairo` — `poseidon_fold`, the 2-input BN254 Poseidon left-fold used for `context`/`scope`.
- `src/interfaces.cairo` — pool + minimal ERC20 interfaces; the flattened `Withdrawal`.
- `src/groth16_verifier*.cairo` — the Garaga-generated withdrawL2 BN254 verifier (proven in the spike).
- `tests/poseidon_parity.cairo` — asserts Garaga Poseidon reproduces the circomlib LeanIMT root.
- `spike/` — the standalone step-1 risk spike (fixture + verifier + reproduce script), kept for reference.

Build: `scarb build`. Test: `snforge test`. Pinned scarb 2.16.1 / starknet-foundry 0.57.0 (`.tool-versions`).

## Step 1 risk spike (reference)

The `spike/` directory holds the risk-first proof that preceded the contract.

## What the spike establishes

The whole design hinges on two things being reproducible off-EVM, in Cairo:

1. **LeanIMT root + nullifier with 2-input BN254 Poseidon.** The L2 tree stores `C_dest` and hashes with Poseidon over **BN254** (via `maci-crypto`), *not* Starknet's native Poseidon (Stark field). `C_dest` can exceed felt252, so Cairo must carry commitments/tree math as `u256`, not `felt252`. This is why the L1 message splits the commitment lo/hi 128.
2. **Groth16 BN254 verification on Starknet** (via Garaga) of the unchanged `withdrawL2` circuit proof.

## Fixture facts (for the Cairo contract)

- Public signals order = `L2ProofLib`: `[existingNullifierHash, noteValue, stateRoot, stateTreeDepth, context]`, `nPublic = 5`.
- `stateRoot` in `public.json` is the LeanIMT root produced by 2-input BN254 Poseidon — the Cairo tree must produce the identical value.
- Leaf = `Poseidon(P.x, P.y, value, Poseidon(ssX, 1))`; nullifier = `Poseidon(stealthPrivateKey, leaf)`.
- vkey/proof: `protocol=groth16`, `curve=bn128`.

## Resolved: context binding is Cairo-friendly, no circuit change

`context` (signal [4]) is **opaque to the circuit** — its only use is `contextSquared <== context * context` (a pinning trick). The circuit requires only that `context < SNARK_SCALAR_FIELD`; it never constrains *how* it's derived. So the derivation is a pool↔SDK convention, freely changeable with **no circuit/zkey/trusted-setup change**.

Decision for the Starknet pool (do NOT reproduce Solidity keccak+abi.encode in Cairo):

```
context = PoseidonBN254([processooor, recipient, feeRecipient, relayFeeBPS, scope])
scope   = PoseidonBN254([pool_addr_felt, starknet_chain_id, asset_felt])
```

- Output is natively a BN254 field element (no mod-p reduction). Reuses the Poseidon the pool needs anyway; no new primitive.
- Flatten the Cairo `Withdrawal` — pass `recipient`/`feeRecipient`/`relayFeeBPS` as felts, drop the nested `bytes data`/`RelayData`.
- SDK: add a `calculateContextStarknet` variant (mirrors the existing `calculateContext` / `calculateWithdrawalContext` split in `packages/sdk/src/crypto.ts`). EVM L2 pool keeps keccak; same circuit serves both.
- Pin field order + arity (optionally a domain tag). All inputs are `< 2^251 < p_bn254`, so no splitting (only `C_dest` splits lo/hi, over the wire).
